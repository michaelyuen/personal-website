overview

Logoji Labs empowers organizations with a unique tool to create one-of-a-kind marketing campaigns. In the lab, it is possible to combine any logo and url to create a custom, clickable logoji. The publish page makes it easy to create posts with these logojis embedded directly into the text, which will be shared on the chosen, connected Facebook Page either immediately or at a scheduled time in the future. Once posted, the feedback page provides insight into the impact of the logojis used by tracking the number of times they have been clicked.

my role

I take pride in all of my work, but this project in particular stands out as my favorite. The reason is because of the level of creative freedom and ownership I was allowed. Everything user-facing as well as the logic behind the scenes that supports it was built by me. While I'm not a traditional designer nor do I have an education in UX, I believe that the experience I have crafted is clean and delightful. I find the interface to be aesthetically pleasing, but more importantly highly functional.

design

material design

In general, I've always preferred flat over realistic design, but there still felt like something was missing. Upon being introduced to Google's Material Design, it was love at first sight. Therefore, I decided that Logoji Labs was a perfect time to put these design specs to work. Not everything was approved, but I believe the essense remains and looks amazing.

always responsive

Responsive design is a given with the prevalence of mobile devices. The application must be able to function consistently well no matter what type of device is being used and the design must be cohesive. While this does present additional challenges and complexity, it becomes quite easy with the use of CSS media queries coupled with Bootstrap's grid system.

By placing the components of the content into individual cards, I've separated the sections logically and visually. This also allows for consistency and flexibility across devices. The screens show how nicely this design conforms to various widths.

content is king

Getting more specific, designing the details of each page was actually rather easy as well. The most important determining factor is ultimately the content that the design needs to support. This is where user experience considerations come into play and provide clarity for the design decisions. The underlying theme was already in place, so the question became less of "what looks good" and more "what works well".

Foremost, this is a service and in order for the design to be successful, it must empower the user's task. In this case, I will discuss some of the desicion making process involved in creating the Publish page, which has by far the most complex content.

Hopefully the intended interaction is obvious, presented in a relatively simple UI that is straight forward. I started by breaking the major functions into tabs, immediately separating the concerns and reducing complexity by half. While everything could be presented to the user immediately, there are several good reasons for not doing so.

First, it is overwhelming. There is a lot to take in from the create section alone and cluttering that with even more interface components would make it even more difficult to focus.

Next, it is unnecessary. I have presented everything the user needs to create a post with their approved logojis, select which connected page it should be posted on, and immediately share it. The task of scheduling the post to be shared at a later date is completely independent, and if that isn't desired then the user will not be forced to see the UI and mentally block it as irrelevant.

Finally, it separates the concerns. For the user, they only need focus on one task at a time; creating the post, then scheduling the post. In this manner they are empowered with a functional design that will enhance their efficiency.

development

In order to create a seamless publishing experience, there is a lot going on under the hood. I've discussed why I'm particularly proud of this application from a design standpoint, but now I would like to explore a few of the most interesting development challenges.

inserting a logoji

This is such a cool feature and it really captures the essense of what I love about front-end development. While the logic behind the scenes is rather complex, it comes together beautifully in creating an experience that doesn't feel complicated at all.

The first step involved is finding the location of the cursor. The user must be able to consistently expect the selected logoji to be inserted where they want, regardless of if that's at the end, the beginning, or anywhere in between. I won't reveal the exact magic, but it involved the use of the Selection and Range Web APIs.

It is worth noting as well that capturing the selection accurately was a bit more complicated due to the event trigger being the click of a logoji. This is because upon clicking, the message loses focus and therefore its cursor data. Programatically focusing the element causes the cursor to be placed at the starting position by default. Thus, I had to implement some additional logic as a workaround to ensure that the cursor's location was preserved accurately.

The second step is to insert the proper html at the cursor position. Unfortunately, textarea elements can't contain html, so I had to find a creative solution. Enter the HTML5 contenteditable attribute, which allows any element to be editable, and for my needs supports the insertion of html! Coupled with the Range API it was then trivial to create an html fragment from the logoji and insert it into the message seamlessly.

scheduling a post

The logoji insertion feature definitely has more of a wow factor for the user, but the scheduling function is the unsung hero of the publishing process. If it isn't clear already, I want the user to only do the work that is absolutely necessary while I do all of the rest. I've intentionally built this date and time selection interface to be as simple and easy to use as possible, even though there is actually a lot of logic happening with each click.

In the preceeding gif, I'm demonstrating one of the numerous considerations that I had to guard against when it came to date validation. There are suggested time bubbles that must be disabled if less than the current time in order to prevent the user from potentially selected a time in the past. However, when the next day is selected, it enables all bubbles because they all represent a date in the future. This presents an interesting edge case where a user may select a future day with a time that is before the time of the current day, then reselect the current day and be left with the same, now invalid, time.

That is just one example, for there are also such cases presented with the custom time selector where the user must be prevented from entering an invalid time, intentionally or accidentally. Other alternatives are possible, including validating the date only once the user clicks the share button, but this would require error feedback for incorrect selections and detract from the seamless experience I wanted to create. My goal is to give users the same type of experience I expect from a service, to get the job done easily and think as little as possible.
